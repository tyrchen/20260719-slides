# 幻灯片“证据优先”重构规格

> 第四部分的顺序、授权地图与 30—90—365 行动计划，已由后续规格
> [`fourth-section-restructure.md`](./fourth-section-restructure.md) 修订；两者冲突时以后者为准。

## 1. 背景

当前版本共 82 页，已经具备完整的叙事、稳定的中文标题、明确的主题配色和 12 张 D3 图表。1280×720 布局审查结果为零页面溢出、零标题溢出。

深度评审发现的主要问题不是视觉风格，而是证据密度不均：Rustack 部分有提交历史、测试和可量化指标；可穿戴和写书部分仍以方法和流程为主。同时，部分页面用方框横排代替了真正的关系表达，几张 D3 页面的关键节点从透明状态开始，导致首帧近乎空白。

## 2. 受众与成功标准

受众是有软件开发经验、已经使用过 AI 编程工具的工程师。修订后，参与者应该能够：

1. 看到三个真实工程对象及其对应的外部证据。
2. 说清 AI 加速的不只是实现，而是构建、验证、学习和留存项目记忆的整个循环。
3. 区分已经测量的事实、从项目得出的判断和尚待验证的假设。
4. 带走一套可以回到自己项目中使用的工作方法，而不是一组口号。

演示层面的验收标准：

- 总页数为 79，仍保留三个空白 Demo 页。
- 所有标题在 1280×720 下保持单行，不小于 34px。
- 零页面溢出、零标题溢出。
- D3 的实体、标签和主要关系在首帧可读；动画只表达方向、累积、回流或状态变化。
- 图表主要标签在 1280×720 截图中的视觉尺寸不小于约 16px；需要更小字号时，必须删减文字而不是继续缩小。
- 通用过程标签使用中文；AWS、S3、BLE、QUIC、Smithy、Rust 等技术标识保留原名。

## 3. 叙事调整

### 3.1 开场

- 首页不再展示“82 SLIDES”，改为参与者能够记住的结构：三个案例、一套方法、一个未来问题。
- 第 3 页从分类卡片改为三个真实产物的可视化证据：Rustack 终端和指标、鞋上数据到手机的训练会话、已构建的书稿页面。
- “AI 工程产能公式”改成“六个短板，任何一个都能拖垮交付”，用串联电路表达最弱环节，避免伪数学感。
- “从助手到系统”用阶梯而不是连续曲线，每一阶同时给出任务单位、上下文寿命和验证方式。

### 3.2 Rustack

- 保留提交历史，增强“前 24 天”区段和服务数增长，避免平均分配的阶段卡片被误读为等长时间。
- Smithy 页面改为分支图：模型进入生成器，展开为多个服务，再由兼容测试回流到规格。
- “一个阶段怎样走到交付”改为真实失败链路：S3 XML 失败 → 回归测试 → 协议规格 → 最小修复 → 后续默认能力。
- Alternator 页面删除“40% → 80%+”仪表盘。改用有明确分母的三轮修复历史，最终显示选定用例 `457 / 457`，并标注测试集。

### 3.3 可穿戴设备

- 保留第 32 页的端到端数据产品闭环，它是本部分的视觉锚点。
- “字段不难，语义才难”改为双鞋时间轴和数据包缺口图，显示时钟漂移、序号丢失、版本和质量标志。
- “平均功耗”改为电流—时间面积图，画出休眠、采样、Flash 写入和 BLE 发射峰值，并明确这是计算原理示意，真实参数必须由电流分析仪测得。
- “标称参数”页改为证据状态图，将已证实、可以仿真和必须现场测量的事项分开。
- 五系统协同 D3 的所有节点首帧可见，只动画数据和反馈路径。

### 3.4 AI 写书

- “知识脉络”改为概念依赖图，表达前置概念、跨章主线和回访关系，不再使用九个相同的流程框。
- “写作约定”改为“类型检查器”视觉：一段文字通过读者、概念、事实、表达四道检查，不合格项返回修改。
- “AI 写书的真实成本”标注为本项目记录，防止被误读为行业平均数据。

### 3.5 统一方法与职业部分

- 删除“项目不同，方法相同”之后的重复页。
- “把一次失败变成下一次起点”使用真正的回路：意图 → 执行 → 外部证据 → 规格 / 测试 / 规则 → 下一次执行。
- “人的工作发生了什么变化”改为时间重心的转移，避免“过去的工程师只编码”这一假对立。
- 两页十条原则合并为一页“五组工程控制”，完整保留原意，但不在现场逐条朗读。
- “构建—衡量—学习飞快转起来”改为自然中文：“AI 让构建、衡量、学习的每一轮更快”。
- “软件产出”页不再将代码行和 PR 称为“过去的标准”，改成“容易统计的产物”和“真正需要负责的结果”。
- 生命周期曲线明确为概念示意，首帧可读，不用定量图表的语气声称预测。
- 价值矩阵标注为项目经验判断，增大坐标与气泡文字，所有气泡首帧可见。
- “领导力与品味”改为“问题选择与工程品味”，避免将顶层能力限定为管理岗位。
- 30 / 90 / 365 路线图的所有节点首帧常驻。
- 想象力页改为发散—收敛回路：问题 → 多个可能 → 可检验假设 → 实验 → 现实修正。
- 最后两页问题合并成一页，标题为“如果效率不再稀缺，我们想放大什么？”，然后进入最终选择页。

## 4. 页数变更

从 82 页减少到 79 页：

1. 删除统一方法部分中与转场页重复的“三个项目，用的是同一套方法”。
2. 两页“十条原则”合并为一页。
3. 两页未来问题合并为一页。

三个 Demo 页继续保留为空白页。

## 5. 视觉与投影规则

- 软件使用蓝色，软硬件使用黄绿色，知识使用橙色，统一方法使用粉色，职业与责任使用紫色，最终反思使用青色。
- 全局边框从低对比的 11% 提高到约 16%；主要图表路径不小于 2px。
- 不增加新的装饰性圆圈。图中的每一个节点都必须表示对象、决策、证据、反馈或记忆。
- 引文继续使用现有统一样式。
- 深色风格保留，但所有关键边框和图表标签必须在投影截图中可分辨。

## 6. 内容与样式边界

- 页面顺序、标题、正文和语义标记放在 `content.js`。
- D3 数据、布局、首帧和动画放在 `charts.js`。
- 通用布局、投影对比度、静态图表和响应式规则放在 `styles.css`。
- 导航、全屏、标题自适应和页面运行时继续留在 `app.js`，本次不在其中写幻灯片内容。

## 7. 完成门禁

1. `node --check content.js charts.js app.js`。
2. `git diff --check`。
3. 确认 `window.SLIDE_CONTENT.length === 79`。
4. 运行 `audit-slide-site.mjs`，要求零溢出、零标题溢出、零过小标题。
5. 截取并人工检查：开场产物、D3 首帧、Rustack 历史、Alternator、功耗、知识依赖、失败回路、价值矩阵、30/90/365、最终页。
6. 测试直接页码链接、键盘导航、全屏、总览和动态页面返回访问。
7. 提交后推送 `main`，确认 GitHub Pages 工作流成功，线上页面返回 HTTP 200。
