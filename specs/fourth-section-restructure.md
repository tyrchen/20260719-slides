# 第四部分重组与重点图表修订规格

## 1. 问题

本轮修改来自 1280×720 实际截图，而不是代码层面的推测：

1. “六个短板”被画成一条串联线路，表达了断路，却没有表达六个因素相乘、任一项接近零都会拖低整体结果。
2. Alternator 页的 `457 / 457` 字号过大，压过了三轮修复过程。
3. “十条原则，可以收成五组工程控制”在前文没有逐条给出十条原则，标题制造了不存在的前置内容。
4. “哪些工作会变便宜，哪些能力会更值钱”混合了自动化、职业价值和薪酬判断；气泡图难以指导工程师决定任务应该怎样授权。
5. 30—90—365 页只有方向，没有明确动作、交付物和检查点。
6. 第四部分同时讨论产出、团队、职业、责任、好奇心和想象力，但页面顺序没有形成一条连续的推理链。

## 2. 受众结果

第四部分结束后，有一定 AI 使用经验的工程师应该能够回答四个问题：

1. AI 改变了软件生产中的什么？
2. 一项任务可以授权给 Agent 到什么程度？
3. 工程师接下来应该补哪些能力？
4. 明天、三个月内和一年内，具体可以做什么？

## 3. 图表修改

### 3.1 六个因素必须呈现“乘积”

- 使用明确的乘号连接六个因素：模型、上下文、任务拆分、反馈、验证、人的判断。
- 右侧输出为“可靠交付”，中间使用等号而不是普通连线。
- 图表不引入虚构百分比。
- 动画轮流压低一个因素，同时削弱输出，用视觉变化说明“任一项接近零，整体也接近零”。
- 首帧必须完整显示公式、因素解释和输出。

### 3.2 Alternator 结果退居证据位置

- `457 / 457` 保持清晰，但字号从主视觉级降低到结果卡级。
- 三轮修复仍是页面主体，结果卡只负责收束。
- 保留“选定用例”的边界说明。

### 3.3 五组工程控制不再声称来自十条原则

- 标题改为“把前面的做法，收成五组工程控制”。
- 五组内容保持：记忆、边界、证据、分工、约束。
- 这一页承担前三个案例的方法归纳，不引入新的计数体系。

### 3.4 用“授权地图”替代职业价值气泡图

页面问题改为：`任务能交多远，要看两件事`。

两条轴：

- 横轴：机器能否独立验证，从低到高。
- 纵轴：失败影响，从低到高。

四个区域：

- 低验证、低影响：小范围试验。
- 高验证、低影响：Agent 可独立执行。
- 高验证、高影响：自动执行，人工批准。
- 低验证、高影响：人来定方向并随时接管。

任务以标签而不是气泡展示，避免文字重叠。图中至少包括模板代码、文档同步、机械迁移、复杂调试、领域建模、架构取舍、验证设计和事故处理。

页面结论：`授权范围可以扩大，责任边界不会自动转移。`

### 3.5 30—90—365 必须可执行

使用三阶段行动板，每个阶段都包含：

- 一个阶段目标；
- 三项具体动作；
- 一个可检查的交付物。

具体内容：

**前 30 天：跑通一个闭环**

- 选择每周都会重复、失败可以复现的真实任务；
- 记录当前耗时、返工次数和常见失败；
- 写入项目上下文，并补一条自动验收。
- 交付物：一个可以重复执行的任务闭环。

**前 90 天：交付一个纵向切片**

- 让 Agent 跨过研究、规格、实现、测试和审查；
- 把至少十次有效修正写成测试、规格或规则；
- 比较使用 AI 前后的周期、缺陷和人工接管点。
- 交付物：一个真实用户或团队正在使用的端到端能力。

**一年内：建立领域系统**

- 在一个领域积累可复用的规格、评估集和工作方法；
- 设计人和 Agent 的权限、审查与异常接管路径；
- 对质量、成本、运行结果和用户后果负责。
- 交付物：团队可以持续使用、结果可以持续验证的工程系统。

## 4. 第四部分的新顺序

第四部分保持 13 页，页数不增加：

1. 转场：当飞轮越转越快，工程师还负责什么？
2. 章节封面：下一站，工程师的未来。
3. 生产单位：从文件数量转向经过验证的新能力。
4. 生命周期：AI 先压缩实现，人守住问题定义和运行反馈。
5. 授权地图：根据可验证性和失败影响决定任务交多远。
6. 团队协作：方向、并行执行、真实反馈和异常升级。
7. 责任边界：实现可以委托，解释、接管和后果不能外包。
8. 能力结构：工具、领域、验证、问题选择。
9. 职业路径：个人贡献者和管理者分别补什么。
10. 行动计划：30—90—365 的动作与交付物。
11. 好奇心：答案变便宜后，什么值得追问。
12. 想象力：把可能收敛成可检验假设。
13. 章节结论：不要和 AI 比谁写得更快。

这条顺序遵循：`工作怎样变 → 任务怎样授权 → 人仍负责什么 → 个人怎样成长 → 为什么还需要好奇心和想象力`。
